# Тема 7 — Вкладені запити, CTE, VIEW (publishing)

---

## Задача 1 — Автори без замовлень (`task01.sql`)
NOT EXISTS перевіряє для кожного автора чи є хоч одне замовлення на його книги. Якщо підзапит нічого не знаходить то автор потрапляє в результат. Якщо список порожній то всі автори мають замовлення

## Задача 2 — Книги з доходом вище середнього (`task02.sql`)
Підзапит у HAVING рахує середню ціну по всій таблиці. Запит групує по книгах і залишає тільки ті де виручка перевищує середній показник

## Задача 3 — CTE + рейтинг у межах жанру (`task03.sql`)
CTE (WITH sales AS)тимчасова таблиця яка існує тільки під час виконання запиту. В ній рахуємо виручку по кожній книзі. Потім RANK() OVER (PARTITION BY Genre) нумерує книги окремо в кожному жанрі почавши від найприбутковішої

## Задача 4 — VIEW (`task04.sql`)
VIEW - це збережений запит в базі даних. Один раз створити через CREATE OR REPLACE VIEW, і далі можна звертатись до нього як до звичайної таблиці. Там є три приклади використання: всі книги, тільки з продажами, топ жанри.

---

## Задача 5 — Висновок

**Прості SELECT** зручні для разових запитів але якщо логіка складна - запит стає нечитабельним і його важко повторно використати

**Вкладені запити** дозволяють фільтрувати по агрегатах або перевіряти наявність зв'язаних записів без JOIN. Мінус те що при великих таблицях можуть бути повільні бо підзапит виконується для кожного рядка

**CTE** читається як звичайний код зверху вниз, легко дебажити по частинах. підходить для аналітичних запитів з кількома кроками

**VIEW** коли один і той самий складний запит потрібен в кількох місцях. Замість того щоб копіпастити - один раз зберігаємо і використовуємо як обичну таблицю. Якщо логіка змінилась - правимо тільки VIEW а не всі запити
